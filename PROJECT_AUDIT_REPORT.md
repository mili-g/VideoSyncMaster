## 架构评估

### 总体判断

当前架构是“可运行、可扩展，但需要治理”的状态。

优点：

- React、Electron、Python 三层分工仍然清晰可辨
- 前端工作流 Hook 已开始模块化
- 后端 action 分发已经形成雏形
- 批处理功能具备基本工程化能力

问题：

- 跨层参数契约分散
- 顶层控制文件仍然过重
- 批处理稳定性很依赖前端状态机和宿主进程管理
- 日志与协议耦合较深

## 业务解耦评估

### 解耦做得较好的部分

- 翻译、配音、事件消费、字幕导入已经按工作流拆分为多个 Hook
- 后端已区分通用 action 分发与 TTS 特化处理
- 输出工件和批处理资源匹配已有专门工具函数支撑

### 解耦仍不充分的部分

- 前端仍直接感知后端 CLI 细节
- 单流程与批处理存在参数构建重复
- 日志输出协议没有独立层
- Electron 主进程同时承担宿主能力和协议解析，边界偏宽

结论：

当前不是“完全未解耦”，而是“已开始解耦，但跨层契约仍未收口”。

## 模型调用与参数治理评估

### 当前优点

- 后端已有统一参数定义入口 `backend/cli_options.py`
- 前端已逐步收敛到少数 Hook 中组织参数
- 模型服务切换路径清晰，Qwen 和 IndexTTS 已有独立逻辑分支

### 当前问题

- `model_dir` 的语义仍不够清晰，翻译模型和 TTS 资源目录边界不够严格
- `localStorage` 中的配置项由多个 Hook 直接读取，容易产生默认值漂移
- 单流程和批处理参数组装存在重复逻辑
- IPC 接口返回类型偏松，难以保证前后端调用稳定性

结论：

参数体系当前“可跑”，但还不够“可演化”。

## 日志与控制台体验评估

### 当前问题

- stdout 同时承担普通日志和协议日志
- Electron 需要通过字符串规则识别 `[PROGRESS]`、`[PARTIAL]` 等标记
- Python `print`、Electron `console`、前端 `console` 混用较多
- 目前还缺少明确的日志分级和分层规则

### 风险

- 一旦输出格式变化，UI 行为可能被影响
- 调试信息和用户状态信息容易混杂
- 故障定位成本会随着功能增加而上升

结论：

当前日志是“可调试”的，但还不是“高质量可观测”的。

## 最终结论

`VideoSyncMaster` 当前已经具备较强的产品能力，但工程体系正处于一个临界点：

- 继续加功能仍然可行
- 但若不先治理参数契约、日志协议和总控文件复杂度，后续维护成本会快速上升

因此，当前最值得优先做的不是继续堆新功能，而是先把“契约、编排、日志”三件事收敛下来。
